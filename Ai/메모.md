## AI 관련 정보 메모

### AI 엔지니어링의 발전

프롬프트 엔지니어링: AI가 이해하기 좋게 대화를 주고받는 방식의 엔지니어링
-> 컨텍스트 엔지니어링: 프로젝트 상황, 배경정보까지 AI에게 전부 알려주는 방식의 엔지니어링
-> MCP + 스킬: MCP(AI가 쓸 수 있는 도구 목록을 미리 정하는 것), 스킬(특정 작업을 할 때 AI가 참고할 가이드 문서)
-> 하네스 엔지니어링: 스킬이 많아지며 AI가 이해해야할 양이 많아 효율이 떨어지는 문제를 해결하기 위해 나옴, 구조적으로 AI가 범할 수 있는 실수를 방지

하네스 엔지니어링: 에이전트가 실수를 할 때, 그 실수를 다시는 반복하지 않도록 엔지니어링 하는 것  
여기서 말하는 하네스란, AI모델이 아닌 모든 것을 의미한다. 예를 들어, MCP, Skills, Hooks 등이 있다.  
AI 모델 + 하네스 = 하네스 엔지니어링  
심지어 ChatGpt를 통해 질문을 하는 것도 하네스라고 할 수 있다. AI모델은 이전의 대화 내용을 기억하고 기반으로 답변을 해준다.

### 하네스가 왜 필요한가?

1. 컨텍스트 부패: 작업이 길어질 수록 컨텍스트 창이 가득차며 앞의 내용을 잊어버리기 시작
    - 엔트로픽 연구팀의 실험에서 컨텍스트 소진(절반만 구현하고 중단), 조기 종료(구현하다가 미완성 상태를 완성 상태라고 선언)를 선언하는 문제를 발견할 수 있었다.
2. 규칙과 울타리: AI가 정보를 다 알아도 엉뚱하게 작동
    - 구조적인 제약을 두지 않으면 AI가 하고싶은 대로 할 수 있다.

- CLAUDE.md
    - 새 세션마다 리셋되는 기억의 문제를 해결할 수 있다. 세션마다 CLAUDE.md 파일에서 정보를 제공
    - 프로젝트에서 꼭 지켜야할 규칙들과 해서는 안되는 행동을 정의
    - 컨텍스트 부패 문제를 해결할 수 있다.
- 훅(Hooks)
    - 코드 저장 직전 자동 실행
    - 에러가 있으면 Claude한테 다시 돌려보낸다. 문제가 있으면 저장자체가 안되도록 한다.
    - 규칙과 울타리 문제를 해결할 수 있다.

### OpenAI 개발자의 AI 활용 실험

1. AGENTS.md : AI 업무 지침서 - 해도되는 일과 절대 하면 안되는 일을 정의
2. CI 게이트: 코드 저장할 때마다 자동 테스트 실행
3. 도구 경계: AI가 접근할 수 있는 범위와 권한 사전 설정
4. 피드백 루프: AI가 직접 코딩 -> 리뷰 -> 규칙 보강 -> 반복

AI Agent 모델보다 하네스를 바꾸는 것이 성능을 더 개선할 수 있다.

### 하네스의 핵심 규칙

1. 컨텍스트 파일
    - AI가 작업 시작 전 가장 먼저 읽는 파일
    - 세세한 설명보다는 항상 적용이 필요한 최소한의 필수 정보만 기입
    - 세부 내용은 다른 파일에 나눠서 필요할 때만 사용할 수 있도록 하기
    - 실패할 때마다 한 줄씩 추가
2. 자동 강제 시스템
    - 린터, 프리커밋 훅 등으로 정해진 규칙을 반드시 지킬 수 있도록 구조 설계
    - 자동 교정 루프(린터 에러 발생 -> 에이전트가 자동 수정)
    - `성공은 조용히, 실패만 시끄럽게`
3. 가비지 컬렉션
    - AI가 만들어놓은 안 좋은 코드를 주기적으로 제거하기
    - 나쁜 패턴을 학습하는 것을 방지
    - 문서가 실제 코드와 달라지진 않았는지, 규칙을 위반한 코드는 없는지, 사용하지 않는 코드가 쌓이진 않는 지 등
    - 실패 -> 새 규칙 -> 하네스 진화 -> 점차 하네스는 정교하게 작동

### Garbage In, Garbage Out

기획 자체가 안좋으면 좋은 하네스를 거쳐도 좋은 결과물이 나올 수 없다.
