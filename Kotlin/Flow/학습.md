## Flow 공부하면서 남기는 메모 또는 글

### 채널

송신자와 수신자 수에 제한이 없으며 전송된 모든 값을 단 한 번만 받을 수 있다.  
채널은 원소를 기다리는 코루틴들을 FIFO 큐로 가지고 있다.  
Channel은 SendChannel과 ReceiveChannel 인터페이스를 구현한 하나의 인터페이스다.  
SendChannel은 원소를 보내거나 채널을 닫는 용도로 사용하고, ReceiveChannel은 원소를 받을 때 사용된다.  
send 함수는 채널의 용량이 다 찼을 때 중단되고 대부분의 채널은 용량이 제한된다.  
receive 함수는 채널에 원소가 없다면 원소가 들어올 때까지 중단된다.  
중단함수가 아닌 함수로 보내거나 받아야한다면 trySend, tryReceive를 사용할 수 있으며 ChannelResult를 반환받는다.  
용량이 제한적인 채널에서만 trySend, tryReceive를 사용해야 하며 버퍼가 없는 랑데뷰 채널에서는 작동하지 않는다.

수신자는 채널이 닫힐 때까지 원소를 받기 위해 for-loop나 consumeEach 함수를 사용해서 수신할 수 있지만 예외가 발생하거나 정상 종료되는 경우 채널을 닫는 것을 깜빡하기 쉽다.  
만약 예외로 인해 코루틴이 원소를 보내는 걸 중단하면 다른 코루틴은 원소를 받기 위해 영원히 기다리게 된다.  
ReceiveChannel을 반환하는 코루틴 빌더인 produce 함수를 사용하면 빌더로 시작된 코루틴이 어떻게 종료되든 상관없이 채널을 닫는다.

#### 채널 타입

- 무제한(Unlimited): 제한이 없는 용량 버퍼를 가진 채널로, send가 중단되지 않는다.
- 버퍼(Buffered): 특정 용량 크기 또는 Channel.BUFFERED(기본값:64)로 설정된 채널이다.
- 랑데뷰(Rendezvous): 용량이 0이거나, Cahnnel.RENDEZVOUS(용량 0)인 채널로, 송신자와 수신자가 만날 때만 원소를 교환한다.
- 융합(Conflated): 버퍼 크기가 1인 Channel.CONFLATED를 가진 채널로, 새로운 원소가 이전 원소를 대체한다.

#### 버퍼 오버플로의 경우

produce 함수에서는 OnBufferOverflow를 설정할 수 없다.

- SUSPEND(기본값): 버퍼가 가득 찼을 때, send 메서드가 중단된다.
- DROP_OLDEST: 버퍼가 가득 찼을 때, 가장 오래된 원소가 제거된다.
- DROP_LATEST: 버퍼가 가득 찼을 때, 가장 최근의 원소가 제거된다.

Channel의 onUndeliveredElement파라미터는 원소가 어떠한 이유로 처리되지 않을 때 호출된다.(채널이 닫히거나 취소되거나, 함수 호출 중 에러 발생 등등)  
주로 채널에서 보낸 자원을 닫을 때 사용한다.

팬아웃: 여러 개의 코루틴이 하나의 채널로부터 원소 받기, 팬인: 여러 개의 코루틴이 하나의 채널로 원소를 전송

### Hot vs Cold

Hot: 데이터를 소비하는 것과 무관하게 원소를 생성하고, 항상 사용 가능한 상태로 여러 번 사용되었을 때 매번 결과를 다시 계산할 필요가 없다.

- 예시로는 Channel이 있다.
- 수신자가 얼마나 많은 지에 대해서 신경 쓰지 않고 원소는 단 한 번만 받을 수 있다.

Cold: 데이터 요청이 있을 때만 작업을 수행하며 아무것도 저장하지 않으며, 무한할 수 있으며 최소한의 연산만 수행하고 메모리를 적게 사용한다.

- 예시로는 Flow, RxJava 스트림이 있다.
- 빌더가 아니면 어떠한 처리도 하지 않으며, flow는 최종 연산이 호출 될 때 원소가 어떻게 생성되어야 하는 지 정의한 것에 불과하다.

### Flow

플로우의 최종 연산은 코루틴을 중단시킨다.  
플로우도 취소가 가능하고 구조화된 동시성을 갖추고 있으며 연산이 실행될 때 부모 코루틴과의 관계가 정립된다.

#### channelFlow

일반 함수이며, 최종 연산으로 시작된다.  
한 번 시작하면 리시버를 기다릴 필요 없이 분리된 코루틴에서 값을 생성하는 점이 채널과 비슷하다.  
주로 여러 개의 값을 독립적으로 계산해야 할 때 사용한다.  
채널의 produce 블록처럼 ProducerScope에서 작동하며 ProducerScope는 CoroutineScope를 구현하고 있어 새로운 코루틴을 시작할 때 사용할 수 있다.

```kotlin
flow {
    withContext(Dispatchers.IO) {
        emit(1)
    }
}
```

이와 같이 flow의 emit을 단일 코루틴이 아닌 다른 코루틴에서 하게 될 경우 예외(IllegalStateException)가 발생한다.  
channelFlow는 내부적으로 Channel을 사용하며, 내부에서 여러 코루틴을 병렬적으로 수행할 수 있다.   
콜백 기반의 이벤트 처리에 용이하고 send, trySend 메서드를 지원하며 모든 자식 코루틴이 종료 상태가 될 때까지 끝나지 않는다.

#### callbackFlow

사용자 클릭이나 활동 변화를 감지해야 하는 이벤트 플로우가 필요하다면, 감지하는 프로세스는 이벤트를 처리하는 프로세스와 독립적이어야 하므로 channelFlow를 사용해도 좋지만 callbackFlow를 사용하는
것이 더 낫다. 또한, 외부 라이브러리 API 호출하는 경우에도 그렇다.
callbackFlow는 콜백 함수를 래핑하는 방식이고 ProducerScope에서 작동한다.  
awaitClose 블록은 채널이 닫힐 때까지 중단되는 함수로 채널이 닫힌 다음 인자로 들어온 함수가 실행된다.  
만약 awaitClose 블록이 없다면 콜백 등록하고나서 코루틴을 바로 종료된다. callbackFlow 빌더의 실행이 끝나고 기다릴 자식 코루틴이 없으면 해당 코루틴은 종료된다.  
awaitClose를 사용하면 코루틴이 종료되는 것을 막고, 채널이 닫힐 때까지 원소를 감지한다.
trySend로 외부 API의 값을 전달함으로써 콜백 환경에서 안전하게 non-blocking으로 값을 전달할 수 있다.  
trySend는 버퍼가 가득 찰 경우 실패를 반환하며 이벤트가 유실될 수 있지만 trySendBlocking은 버퍼에 공간이 생길 때까지 대기하며 이벤트 유실이 없지만 스레드를 블로킹하게 된다. 마치, 코루틴 없이
send 처럼 작동하게 만드는 메서드다.

### Flow 생명주기 함수

플로우가 완료되거나 예외가 발생했을 때, 정보가 전달되어 중단 단계가 종료된다. 모든 정보가 플로우로 전달되므로 값, 예외 및 다른 특정 이벤트를 감지할 수 있다.

#### onEach

중단함수이며, 원소는 순서대로 처리되며 플로우의 값을 하나씩 받기 위해 사용한다.

#### onStart

플로우가 시작되는 경우 리스너를 설정한다.  
첫 번째 원소를 요청하는 시점에 호출되는 함수고, 원소들은 onStart로부터 아래로 흐르게 된다.

#### onCompletion

플로우가 완료(정상, 예외)되었을 때 호출되는 리스너를 설정할 수 있다.

#### onEmpty

upstream flow에서 아무런 값도 내려받지 못한 경우 실행된다. 기본값을 설정하는 목적으로 사용될 수 있다.

```kotlin
flowOf(1, 2, 3)
    .filter { it > 10 }
    .onEmpty { emit(-1) }
    .collect { println(it) }
```

1, 2, 3을 emit 했으나 필터링되어 아무런 값도 수집하지 못해 -1이 출력된다.  
만약 onEmpty 가 filter보다 앞선 파이프라인에 있다면 1, 2, 3 값이 방출되었다고 판단해서 -1이 출력되지 않는다.

#### catch

플로우를 만들거나 처리하는 도중 예외가 발생하면 예외는 아래로 흐르면서 처리하는 단계를 하나씩 닫는다.  
catch는 예외를 잡고 전파되는 것을 멈추게 하며, 이후 새로운 값을 내보내고 남은 플로우를 지속하게 해 별도의 작업을 처리할 수 있다.    
catch는 파이프라인 윗 부분에서 던진 예외에만 반응한다.  
만약 flow에서 예외가 발생했는데 처리하지 못했다면 플로우를 즉시 취소하며, collect는 예외를 다시 던진다.  
중단 함수와 coroutineScope와 동일한 방식으로 예외를 처리한다.  
플로우 바깥에서 try-catch 블록을 사용해서 예외를 잡을 수도 있다.

#### flowOn

플로우 연산과 빌더의 인자로 사용되는 람다식은 모두 중단함수다.  
중단함수는 구조화된 동시성을 위해 컨텍스트가 필요하며 부모-자식 관계를 유지한다.  
flow 함수들의 컨텍스트는 collect가 호출되는 곳의 컨텍스트를 얻어오며 flowOn은 플로우 상위 파이프라인에 있는 함수에서만 작동한다.

#### launchIn

collect는 플로우가 완료될 때까지 코루틴을 중단하는 중단 연산이다.  
launch 빌더로 collect를 래핑하면 플로우를 다른 코루틴에서 처리할 수 있다.  
별도의 코루틴에서 플로우를 시작하기 위해 launchIn을 주로 사용한다.

### Flow 처리(processing)

- map: 새로운 플로우를 생성해 리시버를 통해 변환 함수에 따라 변환하는 함수
- filter: 주어진 조건에 맞는 값들만 가진 플로우를 반환하는 함수
- take: 설정한 개수 만큼의 원소만 통과
- drop: 설정한 개수 만큼 무시하고 원소 받기

#### Flow 처리방식

Flow 객체는 파이프라인 마다 생성되며 상위 flow를 collect 하는 단일 체인 구조로 되어 있다.  
파이프라인에서 호출은 collect를 통해 upstream으로 전파되며, 데이터는 emit이 호출되며 downstream으로 전달된다.

#### Flow 합치기

- merge: 두 개의 플로우에서 생성된 원소들을 하나로 합친다.
  하나의 플로우의 원소가 지연된다고 해서 다른 플로우의 원소 생성이 중단되지 않는다.
- zip: 두 플로우로부터 쌍을 만든다.
  원소가 쌍을 이루는 방법을 정하는 함수가 필요하며, 쌍이 될 원소가 원소가 모두 도착할 때까지 기다려야한다.  
  쌍을 이루지 못하고 남은 원소는 유실되므로 하나의 플로우에서 zipping이 완료되면 생성되는 플로우 또한 완료된다.
- combine: 모든 새로운 원소가 전임자를 대체한다. 첫 번째 쌍이 이미 만들어졌다면 다른 플로우의 이전 원소와 함께 새로운 쌍이 만들어진다.  
  두 플로우 모두 닫힐 때까지 원소를 내보낸다. 주로 두 데이터 소스의 변화를 능동적으로 감지할 때 사용된다.    
  View가 감지 가능한 원소 두 가지 중 하나라도 변경될 때 반응해야 한다면 combine을 주로 사용한다.

- fold: 초기값부터 시작해서 주어진 원소에 대해 두 개의 값을 하나로 합치는 연산을 적용한다. 최종 연산에 해당하며 flow 완료될 때까지 중단된다.
- scan: 누적되는 과정의 모든 값을 생성하는 중간 연산이다. 변경 내역에 대한 객체가 필요할 때 주로 사용된다.

- flatMapConcat: 생성된 플로우를 하나씩 처리한다. 두 번째 플로우는 첫 번째 플로우가 완료되었을 때 시작할 수 있다.
- flatMapMerge: 만들어진 플로우를 동시에 처리한다. concurrency 인자를 사용해 동시 처리할 수 있는 플로우의 수를 설정할 수 있다.(기본값 16)
- flatMapLatest: 새로운 플로우가 나타나면 이 전에 처리하던 플로우는 사라진다.

#### 중복 제거 함수

- distinctUntilChanged: 반복되는 원소가 동일하다고 판단되면 제거한다. 바로 이전의 원소와 동일한 원소만 제거한다.
- distinctUntilChangedBy: 두 원소가 동일한지 판단하기 위해 비교할 key selector를 인자로 받는다.

### SharedFlow & StateFlow

#### SharedFlow

SharedFlow를 통해 메시지를 보내면 대기하고 있는 모든 코루틴이 수신하게 된다.  
replay 인자를 설정하면 마지막 전송한 값들이 인자 값 만큼 저장된다.  
MutableSharedFlow는 SharedFlow와 FlowCollector 모두를 상속한다.  
SharedFlow는 Flow 상속 + 감지하는 목적, FlowCollector는 값을 내보내는 목적으로 사용된다.

- shareIn
  SharedFlow를 만들고 Flow의 원소를 보낸다. 플로우 원소를 모으는 코루틴을 시작하므로 첫 번째 인자로 코루틴 스코프를 받는다.  
  두 번째 인자로 SharingStarted를 설정해 리스너의 수에 따라 값을 언제부터 감지할 지 결정할 수 있으며 세 번째 인자로 replay를 설정할 수 있다.
    - SharingStarted.Eagerly: 즉시 값을 감지하기 시작하고 플로우로 값을 전송한다. replay 값에 제한이 있고 감지 시작하기 전에 값이 나오면 일부 유실할 수 있다.(replay가 0이라면
      먼저 들어온 값은 모두 유실)
    - SharingStarted.Lazily: 첫 번째 구독자가 나올 때 감지하기 시작한다. 첫 번째 구독자는 모든 값을 수신받을 수 있으며, 이후 구독자는 replay 수만큼 가장 최근 값들을 받게 된다.  
      모든 구독자가 사라져도 데이터를 방출하는 Flow는 Activity 상태지만, 구독자가 없으면 replay 수만큼 최근 값들을 캐싱한다.
    - SharingStarted.WhileSubscribed(): 첫 번째 구독자가 나올 때 감지하기 시작하며, 마지막 구독자가 사라지면 Flow도 멈춘다.  
      멈췄을 때 새로운 구독자가 나오면 플로우가 다시 시작된다. stopTimeoutMillis(마지막 구독자 사라지고 나서 감지할 시간), replayExpirationMillis(멈춘 뒤 replay 값을
      가지고 있는 시간)을 가지고 있다.

#### StateFlow

SharedFlow의 개념을 확장시킨 것으로 replay 인자 값이 1인 SharedFlow와 비슷하게 작동한다.    
value 프로퍼티로 접근 가능한 값 하나를 항상 가지고 있으며 초기 값은 생성자를 통해 전달되어야 한다.

- stateIn
  Flow를 StateFlow로 변환하는 함수로, 스코프에서만 호출 가능하지만 중단 함수다.

```kotlin
suspend fun main() = coroutineScope {
    val flow = flowOf("A", "B", "C")
        .onEach { delay(1000) }
        .onEach { println("Produced $it") }
    val stateFlow: StateFlow<String> = flow.stateIn(this)

    println("Listening")
    println(stateFlow.value)
    stateFlow.collect { println("Received $it") }
}
```

stateFlow는 항상 값을 가져야 하기에 명시하지 않았을 경우 첫 번째 값이 계산될 때까지 기다려야한다.  
그래서 위 코드의 경우 flow의 초기값이 없어 stateFlow.value 또는 collect가 호출되는 순간까지 중단상태로 존재하며 stateFlow.value 호출 시점에 초기값을 생성한다.  
Produced A -> Listening -> A -> Received A -> Produced B -> ... -> Received C 순서로 출력된다.

만약, stateIn의 시그니처가 scope, started, initialValue 의 파라미터를 받는 경우에는 초기값이 존재하므로 중단함수가 아니다.

### 사용 예제

- 여러 개의 값을 다루는 경우 Flow를 사용한다. API를 통해 하나의 값을 가져올 때는 중단함수를 사용하는 것이 가장 좋다.
  웹소켓을 설정하고 메시지를 기다릴 때와 같은 상황에서는 Flow를 사용해야 한다.
    - 만약 사용하는 라이브러리에서 값을 반환할 수 없다면 Flow를 만들 때 callbackFlow, channelFlow를 사용한다. 그리고 awaitClose를 반드시 넣어줘야 한다.
- 컬렉션 처리 함수와 async를 함께 사용하면 리스트의 각 원소를 비동기로 처리할 수 있다.
    - 만약 동시성 호출 수를 제한하고 싶다면 Flow의 flatMapMerge를 concurrency 파라미터와 함께 사용할 수 있다.
