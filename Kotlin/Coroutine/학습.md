## 코루틴 공부하면서 남기는 메모 또는 글

### 코루틴의 장점

1. RxJava, 콜백 방식에 비해 비동기로 전환하기 위해 필요한 코드 수정이 적다.
2. 스레드에 비해 컨텍스트 스위칭 비용이 저렴하고 간단하다.
3. 중단함수를 제공해 동시성 프로그래밍을 쉽게 구현할 수 있으며, 스레드를 효율적으로 사용할 수 있다.

### 중단은 어떻게 작동할까?

suspend 키워드가 있으면 컴파일러의 IR 변환 단계에서 Continuation 객체를 함수의 마지막 파라미터에 포함시킨다.  
Continuation 객체는 상태머신 역할을 하며 중단, 재개 지점을 판단할 수 있다.  
Continuation 내부에서 중단 상태인 경우라면 다른 코루틴이 실행될 수 있도록 중단을 판단할 수 있는 값을 반환해야 한다. 그렇기 때문에 Continuation은 Any 또는 Any? 타입을 반환하게 된다.

runBlocking 빌더에서 디스패처를 변경해 다른 스레드에서 작동하게 할 수 있지만, 이 경우에도 코루틴이 완료될 때까지 스레드가 블로킹된다.

launch, async 빌더는 CoroutineScope의 확장함수지만 runBlocking은 기본 함수이기에 루트 함수로만 사용할 수 있다.  
모든 코루틴은 스코프 또는 runBlocking에서 시작된다.

### 코루틴 = 구조화된 동시성 (부모 자식 관계의 중요한 특징)

1. 자식은 부모로부터 컨텍스트를 상속받는다. (코루틴의 기본 특성)
2. 부모는 모든 자식이 작업을 마칠때까지 기다린다.(Job과 연관)
3. 부모 코루틴이 취소되면 자식 코루틴도 취소된다.(Job과 연관)
4. 자식 코루틴에서 에러가 발생하면, 부모 코루틴 또한 에러로 소멸한다.(Job과 연관)

### CoroutineContext

Continuation 객체, CoroutineScope 에서 모두 CoroutineContext를 포함한다.  
CoroutineContext는 Job, CoroutineName, CoroutineDispatcher, CoroutineExceptionHandler 같은 Element 객체들이 인덱싱된 집합이다.  
CoroutineContext에 같은 키를 가진 원소가 있다면 새로운 원소가 기존 원소를 대체한다.  
CoroutineDispatcher의 기본값은 Dispatchers.Default다.

### Job

Job은 코루틴이 상속하지 않는 유일한 CoroutineContext다.
Job은 부모-자식 관계에 의해 부모와 자식 모두를 참조할 수 있다.

지연 시작되는 코루틴은 New 단계에서 시작되며 일반적인 코루틴은 Active 상태에서 시작된다.  
Active 상태에서 Job 실행되고 코루틴은 Job을 수행  
실행이 완료되면 Completing 상태에서 대기하다가 모든 자식들의 실행이 모두 종료되었을 때 completed 상태가 된다.  
실행 중 취소 또는 실패하게 된다면 Cancelling 상태가 되어 자원 정리 작업을 수행할 수 있으며 그 후, cancelled 상태가 된다.

Job을 코루틴 빌더에 생성해서 넣어주게 된다면 구조화된 동시성이 꺠지게 된다.

```kotlin
suspend fun main(): Unit = coroutineScope {
    launch(Job()) {
        // 새로운 Job을 생성해서 넣어주기에 부모-자식관계가 깨져 내부 로직이 실행되지 않는다.
    }
}
```

`join()` : 지정한 Job이 Completed 또는 Cancelled 상태에 도달할 때까지 기다리는 중단 함수

Job 팩토리 함수가 반환하는 타입은 하위 인터페이스 타입인 CompletableJob(2가지 메서드를 추가 제공해 기능성을 확장한 타입) 이다.

```kotlin
public fun Job(parent: Job? = null): CompletableJob
```

- `complete()`: Job을 완료하는 데 사용, 모든 자식 코루틴은 작업 완료될 때까지 실행 상태를 유지하지만, 해당 메서드를 호출한 Job 에서 새로운 코루틴이 시작될 수 없다.

`cancel()` : Job을 취소할 수 있다.

- 호출한 코루틴은 첫 번째 중단점에서 Job을 끝낸다.
- Job이 자식을 가지고 있다면 같이 취소되며, 부모에는 영향을 주지 않는다.
- 취소된 Job은 새로운 코루틴의 부모로 사용될 수 없다. Cancelling 상태가 되었다가 Cancelled 상태로 바뀐다.
- cancel 호출된 후 다음 작업 진행을 위해 join을 호출하는 것이 일반적이다. 호출하지 않을 경우 race condition 발생 가능성 존재
    - join 함수를 호출하면 cancelled 상태를 보장할 때까지 중단되므로 race condition 발생 가능성 제거 가능
- 취소되면 첫 번째 중단점에서 CancellationException 예외를 던진다.
- try-catch에서 CancellationException 을 잡고 후처리 과정을 진행하고 싶어도 Job은 이미 Cancelling 상태가 되었기에 다른 코루틴을 시작하거나 중단하는 것은 불가능하다.
    - 만약 실패 시, 데이터베이스 변경 사항을 롤백해야 하는 경우와 같이 이미 취소된 코루틴 내에서 중단 함수를 호출해야 하는 경우, withContext(NonCancellable)로 포장해서 중단함수를
      호출할 수
      있다.
- invokeOnCompletion 메서드를 호출하면 Job이 Completed 또는 Cancelled 상태에 도달했을 때 필요한 작업을 처리할 수 있다.
    - Job이 예외 없이 끝나면 null을 반환하며, 코루틴이 취소되었다면 CancellationException을 반환하며, 그 외 코루틴을 종료시킨 예외를 반환한다.
    - 취소하는 중에 동기적으로 호출되며, 어떤 스레드에서 실행할지 결정할 수 없다.

### 코루틴 예외처리

코루틴은 예외를 받았을 때 본인을 취소하고 예외를 부모로 전파한다.  
부모는 자기 자신과 자식들 모두 취소하고 예외를 부모로 전파하고 더 이상 부모가 없다면 프로그램을 종료시킨다.

SupervisorJob을 사용하면 자식에서 발생한 모든 예외를 무시할 수 있다.

CancellationException은 부모까지 전파되지 않는다. 단지, 현재 코루틴을 취소 시킬 뿐이다.

### 코루틴 스코프

GlobalScope는 EmptyCoroutineContext를 가진 스코프다.

- 해당 스코프에서 async를 호출할 경우 취소될 수 없으며, 부모로부터 스코프를 상속받지 않는다.
- 결과적으로 부모가 취소되어도 async 내부 함수가 실행중인 상태가 될 수 있어 메모리 누수가 발생할 수 있다.

코루틴 스코프 함수는 중단함수에서 코루틴 스코프를 만들기 위해 사용되는 함수다.

- 스코프 함수로는 coroutineScope, supervisorScope, withContext, withTimeout 함수가 있다.
- 중단함수라는 특징이 있으며 중단 함수의 Continuation 객체의 CoroutineContext를 사용한다.
- 일반 함수와 동일한 방식으로 예외를 던진다.

#### coroutineScope

- 새로운 스코프를 시작하는 중단함수면서, 인자로 들어온 함수가 생성한 값을 반환한다.
- 새로운 코루틴이 끝날 때까지 coroutineScope를 호출한 코루틴을 중단하기에 호출한 코루틴이 작업을 동시에 시작하지는 않는다.
- 부모로부터 컨텍스트를 상속받으며, 자신의 작업을 끝내기 전까지 모든 자식을 기다리며 부모가 취소되면 자식들 모두를 취소한다.
- 코루틴 빌더와 달리 스코프에 속한 자식에서 예외가 발생하면 다른 모든 자식이 취소되고 예외가 다시 던져진다.

#### withContext

- Scope의 CoroutineContext를 변경할 수 있다.

#### supervisorScope

- SupervisorJob을 가진 스코프로 자식 코루틴에 예외를 전파하지 않는다. 주로 서로 독립적인 작업을 시작하는 함수에서 사용된다.

#### withTimeout

- 인자로 들어온 람다식을 실행할 때 시간 제한을 설정할 수 있다.
- 시간 초과 발생 시, TimeoutCancellationException(CancellationException 서브타입) 던지며 해당 코루틴만 취소되고 부모에서 잡히며 자식들만 취소하고 다른 코루틴들에게는
  영향을 주지 않는다.
- withTimeoutOrNull 로 예외가 아닌 null을 반환하게 할 수도 있다.

### 디스패처

디스패처는 코루틴이 실행되어야 할 스레드, 스레드 풀을 결정할 수 있도록 한다.

#### Default Dispatcher

- CPU 집약적 연산을 수행하도록 설계되었다.
- 컴퓨터의 CPU 개수와 동일한 수의 스레드 풀을 가지고 있으며 비용이 많이 드는 작업이 Default 디스패처의 스레드를 다 사용하고 있어서 다른 코루틴이 실행되지 못하는 것 같은 경우
  limitedParallelism을 설정하여 디스패처가 특정 수 이상의 스레드 사용을 제한할 수 있다.

#### Main Dispatcher

- 안드로이드에서 UI 와 상호작용하는 데 사용하는 단일 스레드

#### I/O Dispatcher

- 파일을 읽고 쓰거나, shared preference 사용, 블로킹 함수 호출하는 경우 등 I/O 연산으로 스레드를 블로킹하는 경우를 고려해 설계되었다.
- 64(더 많은 코어라면 해당 코어의 수)로 제한이 된다.
- Dispatchers.IO는 Default 와 같은 스레드 풀을 공유한다.
- 최대 활성화 가능 스레드 수는 Dispatchers.IO(64개)+CPU 코어수이고, 스레드 풀의 모든 스레드들이 블로킹 된다면 이후 요청은 모두 대기하게 된다.
- limitedParallelism 로 지정할 경우 독립적인 스레드 풀을 가진 새로운 디스패처를 만든다.

Dispatchers.Default의 limitedParallelism은 디스패처에 스레드 수 제한을 추가하는 반면, Dispatchers.IO는 독립적인 디스패처를 만든다.

#### Unconfined Dispatcher

- 스레드를 바꾸지 않는 특징이 있다.
- 시작한 스레드에서 실행되고 재개되었을 때는 재개한 스레드에서 실행이 된다.

#### Dispatchers.Main.immediate

- 현재 디스패처와 동일한 디스패처를 갖고 있는 함수를 호출한다면 불필요한 재배정 비용이 발생하게 된다.
- 대기 큐가 쌓여있다면 차례가 오기까지 지연이 발생할 수 있다.
- 이러한 경우를 방지하기 위해 불필요한 dispatch를 생략하는 Dispatchers.Main.immediate가 있다.
- 이미 Main Dispatcher 라면 대기 큐 생략, 다른 Dispatcher 였다면 대기 큐에 차례를 기다리고나서 Dispatchers.Main 으로 전환

### 공유 상태 문제

자바에서 사용되는 synchronized 블록이나 동기화된 컬렉션을 활용해서 동시성 문제를 해결하면 어떤 문제가 있을까?

- synchronized 블록 내부에서는 중단 함수를 사용할 수 없다.
- 그렇기에 코루틴이 자기 차례를 기다릴 때 스레드를 블로킹한다.
- 메인 스레드가 블로킹 되거나, 하나의 함수를 처리하기 위해 스레드를 모두 사용해버리는 상황이 발생하거나 자원이 낭비되는 등의 문제가 발생할 수 있다.

Java에서 원자성을 보장하기 위한 객체를 제공하며 이는 lock 없이 CAS 방식으로 원자성을 보장하기에 효율적일 수 있다.  
하지만 하나의 연산에서 원자성을 보장한다고 전체 연산에서 원자성이 보장되는 것은 아니다.

#### 싱글 스레드로 제한된 디스패처

공유 상태 관련한 대부분의 문제를 해결할 수 있는 가장 쉬운 방법이다.

1. 코스 그레인드 스레드 한정: 함수 전체를 단일 스레드로 처리
    - 사용하기 쉽고 충돌을 방지할 수 있지만, 함수 전체에서 멀티스레딩의 이점을 누릴수는 없다.
2. 파인 그레인드 스레드 한정: 상태 변경하는 구문들만 래핑하는 방법
    - 조금 더 번거롭지만, critical section이 아닌 부분이 블로킹되지 않으며 CPU 집약적인 경우에 더 나은 성능을 제공한다.

#### 뮤텍스

뮤텍스의 가장 중요한 기능은 lock이다.

- 첫 번째 코루틴이 lock을 호출하면 열쇠를 가지고 중단 없이 작업을 수행하고 다른 코루틴이 lock을 호출하면 첫 번째 코루틴이 unlock을 호출할 때까지 중단된다.
- lock, unlock을 수동으로 처리할 경우 unlock을 호출하지 않는 deadlock이 발생할 수 있으며, withLock 함수를 통해 블록 내에서 어떤 예외가 발생하더라도 unlock을 실행하게 할 수
  있다.
- synchronized 블록과 달리 스레드가 아닌 코루틴을 중단시킨다.
- 뮤텍스 사용할 때 주의할 점은 코루틴이 lock을 두 번 통과할 수 없다.(lock을 중첩으로 사용할 경우 교착상태 발생)
- 또 다른 주의할 점으로는 코루틴이 중단되었을 때 뮤텍스를 풀 수 없다.
- 하나의 함수에서 여러 중단 함수 호출이 필요한 경우 싱글스레드로 제한된 디스패처를 사용하는 방식을 고려해보기

#### 세마포어

뮤텍스와 비슷한 방식으로 작동하지만 둘 이상 접근할 수 있다.  
여러 개의 접근을 허용하므로 공유 상태로 인해 생기는 문제를 해결할 수는 없지만 동시 요청을 처리하는 수를 제한할 때 사용할 수 있다.

### 코루틴 테스트

코루틴에서는 가상 시간동안 시간을 흘러가게 하여 실제 상황과 유사하게 테스트할 수 있다.  
TestCoroutineScheduler를 사용하려면, 지원하는 디스패처를 사용해야 한다. 일반적으로 StandardTestDispatcher를 사용한다.  
테스트 디스패처로 시작된 코루틴은 가상 시간만큼 진행되기 전까지 실행되지 않는다.  
일반적으로 가상 시간이 흐를 동안 호출될 모든 작업을 실행하는 advanceUntilIdle과 advanceTimeBy에 일정 밀리초를 넣어 그동안의 실행될 모든 연산을 수행하게 하는 방식이 있다.  
StandardTestDispatcher를 스코프로 래핑해서 사용하는 방식 대신 동일한 역할을 수행하는 TestScope를 사용할 수 있다.    
스코프가 사용하는 스케줄러에 advanceUntilIdle, advanceTimeBy, currentTime 위임된 프로퍼티가 존재한다.

#### runTest

TestScope에서 코루틴을 시작하고 즉시 유휴 상태가 될 때까지 시간을 흐르게 한다.  
자식 코루틴이 끝날 때까지 기다린다. 절대 끝나지 않는 프로세스를 시작한다면 테스트도 종료되지 않는다. 이런 경우를 위해 backgroundScope를 제공한다.

### 사용 예제

- 코틀린 코루틴을 지원하지 않는 라이브러리를 사용해 콜백함수를 사용해야 하는 경우라면 suspendCancellableCoroutine을 사용해 콜백 함수를 중단 함수로 변환한다.
- 블로킹 함수를 사용할 수 밖에 없는 경우에 일반적인 중단 함수에서 블로킹 함수를 호출해서는 안된다.
    - 코틀린 코루틴에서 스레드를 정밀하게 사용하기에 스레드가 블로킹되면 심각한 문제를 불러일으킬 수 있다.
    - Dispatcher를 명시하지 않고 블로킹 함수를 호출해서는 안되며, withContext를 사용해 디스패처를 명시한다.
- 인라인 람다식 함수라면 suspend 제어자가 없어도 중단함수를 호출할 수 있다.
- withContext(EmptyCoroutineContext) 대신 coroutineScope를 사용하자
- awaitAll을 사용하자, 비동기 작업 중 하나가 예외를 던졌을 경우 곧바로 멈춘다.
- 무거운 함수에서는 yield를 사용해서 오랜시간 점유하지 않도록 해 다른 프로세스를 기아상태로 만드는 일을 없게 만들자.
- 스코프가 취소되고 나면 취소된 스코프를 재사용할 수 없다. 스코프의 모든 작업을 취소하지만 스코프를 Active 상태로 유지하고 싶다면 스코프의 모든 자식들만 취소한다.
