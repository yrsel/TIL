# Compose deep dive

# Compose Runtime

- Composition이 생성되면 Composer는 스스로 구축한다.
    - Composer는 currentComposer 매커니즘을 통해 접근할 수 있으며, Composition이 관리하는 트리를 생성하고 업데이트하는 데 사용된다.
- Jetpack Compose Runtime에 접근할 수 있게 하는 진입점
    - Composable 함수 작성: Composable 함수를 작성하는 것만으로 관련된 모든 정보를 방출하고, 해당 함수들에 대한 정보를 런타임과 연결합니다.
        - 다만, Composition 프로세스 없이는 결코 실행될 수 없습니다.
    - setContent: 대상 플랫폼과의 통합 계층이며, 이를 통해 Composition이 생성되고 시작됩니다.

## 안드로이드에서의 Composition 생성

ViewGroup의 setContent 함수 호출로 새로운 composition을 생성할 수 있습니다.  
내부 코드를 분석해보면, Composition을 AndroidComposeView에 연결하는 방법을 알고 있는 데코레이터인 WrappedComposition이 안드로이드 뷰 시스템에 직접 연결합니다.  
또한, Composition에 전달된 UiApplier 인스턴스가 트리의 루트 LayoutNode를 가리키는 것으로 시작됩니다.

Subcomposition의 경우 상위 composition은 다른 composition이 됩니다. 루트 composition이 경우 상위는 Recomposer가 됩니다.  
그러나 이와 관련 없이 초기 composition을 수행하는 로직은 어떤 경우든 항상 Recomposer에 의존합니다.  
이는 Subcomposition의 경우 루트 composition에 도달할 때까지 지속적으로 composeInitial 호출을 상위 요소에 위임하기 때문입니다.  
composeInitial에서는 다음과 같은 중요한 작업을 수행합니다.

- 모든 State 객체의 현재 값을 스냅샷으로 저장합니다. 이 스냅샵은 가변적이지만 동시성에 대한 안전성을 갖추고 있습니다.
    - State 객체에 대한 모든 변경 사항은 해당 State 객체에 대해서만 발생하고 이후 단계에서 모든 변경 사항을 전역 공유 상태와 원자적으로 동기화하므로, 다른 기존 State 스냅샷에 영향을 주지 않고
      안전하게 수정할 수 있습니다.
    - 변경 가능한 스냅샷의 State 값들은 `snapshot.enter(block: () -> T)`를 호출할 때 전달된 블록에서만 수정될 수 있습니다.
    - 스냅샷을 취할 때, Recomposer는 언급된 State 객체들에 대해 읽거나 쓰기 위한 관찰자들을 전달하며, 이러한 작업이 발생할 때 composition에 알릴 수 있습니다.
    - composition.composeContent(content)와 같은 블록에 snapshot.enter(block) 블록을 전달함으로써 스냅샷에 들어갑니다.
        - 여기가 composition이 실제로 일어나느 곳입니다.
        - 스냅샷에 들어가는 행동은 Recomposer에게 composition 중에 읽거나 쓴 모든 State 객체들이 추적될 것임을 알려줍니다.(composition에 알림)
    - Composition 프로세스는 Composer에게 위임됩니다.
    - Composition이 완료되면, State 객체의 모든 변경 사항은 현재 State 스냅샷에만 적용되므로, 해당 변경 사항을 전역 상태에 전파할 시간이며, snapshot.apply()를 통해
      이루어집니다.

- Composer에게 위임된 실제 Composition 과정들은 다음과 같이 진행됩니다.
    - Composition이 이미 진행 중인 경우 새로 시작할 수 없습니다. 새로운 composition은 폐기되고 예외가 발생합니다.
        - 새로운 composition에 재진입하는 것은 지원되지 않습니다.
    - 보류중인 invalidation이 있는 경우, 해당 invalidation들을 Composer가 유지하는 invalidation 목록으로 복사하여 RecomposeScope의 invalidation을
      기다립니다.
    - Composition이 시작될 예정이므로 isComposing 플래그를 true로 변경합니다.
    - Composition을 시작하기 위해 startRoot()를 호출하면, 이는 슬롯 테이블에서 composition의 루트 그룹을 시작하고 다른 필요한 필드와 구조를 초기화합니다.
    - 슬롯 테이블에서 content의 그룹을 시작하기 위해 startGroup을 호출합니다.
    - content 람다식을 호출하여 모든 변경 사항을 방출합니다.
    - 슬롯 테이블에서 그룹을 끝내기 위해 endGroup을 호출합니다.
    - endRoot()를 호출하여 composition을 끝냅니다.
    - Composition이 완료되었으므로 isComposing 플래그를 false로 변경합니다.
    - 임시 데이터를 유지하는 다른 구조들을 삭제합니다.

### 초기 Composition 후 변경 사항 적용

Applier는 composition.applyChanges() 프로세스 중 기록된 모든 변경 사항을 적용하라는 알림을 받습니다.  
이 작업은 applier.onBeginChanges()를 호출하는 Composition을 통해 수행되며 모든 변경 사항을 실행하고 필요한 Applier 및 SlotWriter 인스턴스를 각 변경 사항에 전달합니다.
마지막으로, 모든 변경 사항이 적용된 후, applier.onEndChanges()를 호출합니다.

그런 다음 등록된 모든 RememberedObserver를 전달하므로 RememberObserver 계약을 구현하는 모든 클래스는 composition에 들어가거나 나갈 때 알림을 받을 수 있습니다.  
LaunchedEffect, DisposableEffect 같은 항목이 RememberObserver를 구현하므로 composition 내의 Composable 수명 주기에 대한 이펙트를 제한할 수 있습니다.

### Composition에 대한 추가 정보

Compose Runtime은 ControlledComposition이라는 composition의 변형에 의존하는데, 이는 외부에서 제어할 수 있도록 몇 가지 추가적인 기능을 제공합니다.  
이를 통해 Recomposer는 invalidation과 추가적인 recomposition을 조율할 수 있습니다.  
composeContent나 recompose 같은 기능이 이에 대한 좋은 사례입니다. Recomposer는 필요할 때 composition에서 이와 같은 작업을 트리거할 수 있습니다.  
Composition은 자신이 관찰하는 일련의 객체들을 감지할 수 있는 방법을 제공하는데, 관찰 중인 객체들이 변경될 때 recomposition을 강제할 수 있습니다.  
예를 들어, 상위 composition에서 CompositionLocal이 변경되면 하위 composition에서 recomposition을 강제하기 위해 Recomposer가 해당 방법을 사용합니다. 이러한 경우
composition들은 상위 CompositionContext를 통해 연결됩니다.

## Recomposer

Recomposer는 ControlledComposition을 제어하고, 궁극적으로 변경 사항들을 적용하기 위해 필요할 때 recomposition을 트리거합니다.  
또한 composition을 시작하거나 recomposition할 스레드 및 변경 사항을 적용할 때 필요한 스레드를 결정합니다.

### Recomposer 생성

해당 진입점은 플랫폼과 Compose Runtime 사이의 연결점이며, 클라이언트 라이브러리에 의해 제공됩니다.  
안드로이드의 경우 Compose UI입니다. composition(내부적으로 자체 Composer 생성)과 그 부모로 사용할 Recomposer를 생성합니다.  
WindowRecomposerFactory는 현재 연결중인 Window에 대한 Recomposer를 생성합니다.

- Compose UI가 Android 수명 주기에 연결된 Recomposer를 생성하는 수단

```kotlin
fun interface WindowRecomposerFactory {
    fun createRecomposer(windowRootView: View): Recomposer

    companion object {
        val LifecycleAware: WindowRecomposerFactory = WindowRecomposerFactory { rootView ->
            rootView.createLifecycleAwareViewTreeRecomposer()
        }
    }
}
```   

createRecomposer를 호출하려면 루트 뷰에 대한 참조를 전달해야 합니다.  
이는 생성된 Recomposer는 수명 주기를 인식하므로 뷰 계층 구조의 루트에 있는 ViewTreeLifecycleOwner에 연결된다는 것을 의미합니다.  
뷰 트리가 연결되지 않은 경우 Recomposer를 취소할 수 있으며, recomposition 프로세스의 누수를 방지하는 데 중요합니다.  
팩토리 함수가 하는 첫 번재 일은 PausableMonotonicFrameClock을 생성하는 것 입니다.

- AndroidUiDispatcher의 모노리틱 클럭(OS나 하드웨어가 직접 계산하는 시간) 위에 수동으로 withFrameNanos 이벤트의 전달을 일시 중지할 수 있는 기능을 확장하는 래퍼입니다.
- 특정 시간동안 프레임이 생성되어서는 안되는 경우에 유용합니다.(UI를 호스팅 하는 Window가 더이상 보이지 않을 때와 같은 경우)
- MonotonicFrameClock은 기본적으로 CoroutineContext.Element를 상속하고 있기 때문에, 다른 CoroutineContext와 결합될 수 있습니다.

```kotlin
val contextWithClock = currentThreadContext + (pausableClock ?: EmptyCoroutineContext)
val recomposer = Recomposer(effectCoroutineContext = contextWithClock)
```

contextWithClock 선언한 Context 변수는 Recomposer가 내부 Job을 생성하여 Recomposer가 종료할 때 모든 composition 또는 recomposition에 사용되는 이펙트들이
취소될 수 있도록 합니다.  
예를 들어, Android의 Window가 파괴되거나 연결되지 않을 때 필요합니다.  
`이 컨텍스트는 composition 및 recomposition 후 변경 사항을 적용하는 데 사용되는 컨텍스트이며,` LaunchedEffect에서 이펙트를 실행하는 데 사용되는 디폴트 컨텍스트이기도 합니다.  
변경 사항을 적용하는 데 사용하는 스레드와 동일한 스레드에서 이펙트를 시작하도록 합니다. Android에서는 일반적으로 메인스레드가 됩니다.

## Recomposition 프로세스

recompsoer.runRecomposeAndApplyChanges() 함수가 호출되면 invalidation을 기다리기 시작하고, invalidation이 발생하면 자동적으로 recomposition 됩니다.

recomposer.runRecomposeAndApplyChanges()를 호출할 때 가장 먼저하는 일은, 해당 변경 사항의 전파 프로세스에 관찰자를 등록하는 것입니다.  
이에 따라 해당 관찰자가 활성화되어 모든 변경사항을 스냅샷 invalidation 목록에 추가하고, 알고 있는 모든 Composer에게 전파합니다.  
이 관찰자는 상태 변경 시 자동적으로 recomposition을 트리거하는 기축입니다.  
스냅샷을 적용하기 위해 관찰자를 등록한 후, Recomposer는 모든 Composition에 대하여 invalidation을 수행하고, 모든 것이 변경되었다고 가정하는 것으로 시작합니다.  
해당 시점 이전에 발생한 모든 변경 사항은 추적되지 않았기 때문에, 이는 invalidation이 수행된 시간을 기점으로 처음부터 기록을 시작합니다.  
그 이후 모든 보류중인 상태 스냅샷의 invalidation 또는 RecomposeScope에서 오는 모든 composition invalidation이 유입되는 것을 마칠 때까지 작업은 지연됩니다.  
그 다음에는 Recomposer 생성 시 제공된 모노리틱 클럭을 사용하여 `parentFrameClock.withFrameNanos{}`를 호출하고, 다음 프레임을 기다립니다.  
나머지 작업은 해당 시간 이후에 수행되는 것들만 처리하는데, 그 목적은 프레임에 대한 변경 사항을 통합하기 위함입니다.  
parentFrameClock.withFrameNanos{} 블록 내에서 Recomposer는 잠재적인 대기자에 대해 먼저 모노리틱 클럭 프레임을 전달합니다. 이로 인해 추적이 필요해지고, 그 결과로 새로운
invalidation이 발생할 수 있습니다.  
그 다음 Recomposer는 보류 중인 모든 스냅샷에 대해 invalidation을 수행하는데, 마지막 recomposition 호출 이후 수정된 모든 상태값을 가져와 Composer의 모든 변경 사항을 보류중인
recomposition으로 기록합니다.  
가령, 상태가 Composable 람다식에 작성된 경우, composition.invalidate()를 통해 invalidation된 composition이 존재할 수도 있습니다.  
Recomposer는 각각에 대해 recomposition을 수행하고 변경 사항의 적용을 보류중인 composition 목록에 이를 추가합니다.

Recomposition이란 Composition 상태(슬롯 테이블)와 구체화된 트리(Applier)에 필요한 모든 변경(Change)을 다시 계산하는 것을 의미합니다.  
Recomposition은 모든 코드를 재사용하기 때문에, 해당 프로세스가 모든 과정을 다시 반복할 필요가 없습니다.  
나중에 composition에 의한 값 변경으로 인해 다시 composition을 수행해야 하는 잠재적으로 따라올 recomposition이 발생할 경우들을 찾아내고, 이를 recomposition을 수행하기 위한
작업으로 예약합니다.  
가령, 상위 항목에서 CompositionLocal에 업데이트가 발생했고, 유효한 하위 composition에서 해당 값을 읽어 들이고 있는 경우 이와 같은 작업들이 발생할 수 있습니다.    
마지막으로, 적용할 변경 사항이 있는 모든 Composition들을 검토하고 해당 Composition에 대해 composition.applyChanges()를 호출합니다. 그 후에 Recomposer의 상태 또한
업데이트 합니다.

# Compose UI

## Compose UI와 Compose Runtime의 통합

Compose Runtime과 UI를 통합할 때의 목표는 사용자가 마주할 화면의 레이아웃 트리를 구축하는 것입니다.  
이 트리는 Composable 함수를 실행함으로써 만들어지고 나중에 업데이트됩니다. 트리에 사용되는 노드 타입은 Compose UI에서만 알려져 있으므로 런타임은 이에 대해 알지 못합니다.  
Compose UI는 KMP 프레임워크이긴 하지만, Compose UI의 노드 타입은 현재 Android와 Desktop에서만 사용됩니다.  
Compose for Web과 같은 다른 라이브러리는 다른 노드 타입을 사용합니다.  
이러한 이유로, 런타임은 트리에서 노드를 삽입, 제거, 이동 또는 교체하는 작업을 클라이언트 라이브러리에 위임합니다.

Composition과 recomposition 과정 중 Composable 함수가 실행될 때, Composable 함수는 자신의 변경 사항을 방출합니다.  
이와 함께 Composition이라는 side table이 사용됩니다.   
이 테이블은 Composable 함수의 실행을 실제 노드 트리의 변경으로 매핑하는 것과 관련된 정보를 가지고 있습니다.  
Compose UI를 사용하는 애플리케이션에서는 우리가 표현해야 할 노드 트리의 수만큼 많은 Composition을 가질 수 있습니다.(Activity,Fragment 에서의 setContent 뿐만 아니라,
ComposeView(context) 또는 xml에 ComposeView를 통해서도 Composition을 생성할 수 있따.)

`setContent 함수는 새로운 루트 Composition을 생성하고, 가능한 한 재사용합니다.`  
루트 Composition이라 부르는 이유는 각각 독립적인 Composable 트리를 호스팅하기 때문입니다. 이 Composition 들은 서로 연결되어 있지 않습니다.   
레이아웃 계층을 생성하기 위해 연관된 Composer가 composition 과정을 실행합니다. 이 과정에서 setContent안에 있는 모든 Composable 함수들이 실행되며 변경 사항이 방출됩니다.  
Compose UI에서 이러한 변경 사항들은 UI 노드들을 삽입, 이동, 교체하며 Box, Column, LazyColumn 같은 일반적인 UI 요소에 의해 발생됩니다.  
이런 컴포저블들은 읿나적으로 foundation.material에 속하지만, 결국 모두 Layout(compose-ui)으로 정의되므로 같은 노드 타입, LayoutNode를 방출한다고 볼 수 있습니다.

## Compose UI 관점에서의 Subcomposition

Composition은 Composable 트리의 더 깊은 수준에서 생성될 수 있으며 부모 Composition과 연결될 수 있습니다.  
이를 Subcomposition이라고 부릅니다.  
Composition들은 트리처럼 연결될 수 있습니다. 즉, 각 Composition은 부모 Composition을 나타내는 부모의 CompositionContext에 대한 참조를 가지고 있습니다.(루트
Composition은 예외로, 그 부모는 Recomposer 자체입니다.)

Compose UI에서 Subcomposition을 만드는 이유는 크게 2가지가 있습니다.

1. 초기 composition 과정에서 특정 정보를 알 때까지 연기하기 위해
2. 하위 트리에서 생성되는 노드의 타입을 변경하기 위해

### 초기 composition 과정의 지연

- SubcomposeLayout 을 예로 들 수 있습니다.
    - SubcomposeLayout은 루트가 compose 되는 시점이 아니라 레이아웃 단계에 초기 composition을 수행하도록 결정합니다.
- 이는 Layout과 유사하지만 레이아웃 단계에서 독립적인 Composition을 생성하고 실행합니다.
- 이를 통해 자식 Composable들로 하여금 자신 안에서 계산된 값에 의존하도록 할 수 있습니다.
    - SubcomposeLayout에 대한 사용 예시는 BoxWithConstraints가 있습니다.
    - 부모의 제약사항을 content block에 제공하여 그 값에 따라 내용을 다르게 구성할 수 있습니다.

### 서브 트리의 노드 타입 변경

Compose UI에서 좋은 예시로, 벡터 그래픽을 생성하고 표시하는 것이 있습니다.(rememberVectorPainter)  
벡터 Composable은 벡터 그래픽을 트리로 모델링하기 위해 자체 Subcomposition을 생성합니다.

- Vector Composable이 compose 될 때, VNode라는 다른 노드 타입으로 Subcomposition을 구성합니다.(VNode는 재귀적인 타입으로, 단일 경로나 경로 그룹을 모델링합니다.)
- Icon, Image 같은 Composable 내에서 VectorPainter를 사용합니다.
    - 이는 벡터를 포함하는 Composable이 Layout이므로 관련된 Composition에 LayoutNode를 방출하는 것과 동시에 VectorPainter는 벡터를 표현하기 위한 자체적인
      Subcomposition을 생성하고, 이를 이전 Composition에 연결하여 부모로 만듭니다.
- Vector는 Subcomposition을 통해 모델링 되는데, 이는 벡터 Composable 호출(rememberVectorPainter) 내에서 부모 Composition에서 사용 가능한
  CompositionLocal 중 일부에 접근하는 것이 편리하기 때문입니다.(dp로 주어진 벡터의 사이즈를 px로 변환하기 위해 LocalDensity.current를 사용하고 있습니다.)

## 다양한 타입의 Applier들

Applier를 런타임이 트리의 변경사항을 구체화하기 위해 의존하는 추상화를 제공합니다.(라이브러리가 사용되는 플랫폼에 대해 런타임이 알지 못하도록 의존성을 역전 시킵니다.)
`Applier 인터페이스`와 이를 구현하는 `AbstractApplier 추상클래스`가 있습니다.  
AbstractApplier는 Compose Runtime에서 다양한 Applier간의 공통 로직을 공유하기 위해 제공되는 기본 구현체입니다. 방문한 노드를 스택에 저장하며 현재 방문중인 노드에 대한 참조를
유지합니다.  
AbstractApplier에 스택과 down, up 작업이 포함되어 있어, 자식 Applier들이 노드 타입에 관계없이 동일한 탐색 로직을 공유할 수 있습니다.  
이는 노드 간 부모와 자식 관계를 제공하여, 트리를 탐색하기 위한 관계를 유지하는데 기술적으로 특정 노드 타입이 필요하지 않도록 해줍니다.

Compose UI는 Jetpack Compose Runtime과 안드로이드 플랫폼을 통합하기 위해 AbstractApplier의 두가지 구현을 제공합니다.
방문하는 노드의 유형에 따라 다른 Applier 구현이 사용됩니다. 예를 들어 LayoutNode로 구성된 루트 Composition과 VNode로 구성된 Subcomposition을 가지고 있다면, 전체 UI
트리를 구체화하기 위해 두 Applier가 모두 사용될 것입니다.

- UiApplier: 대부분의 안드로이드 UI를 렌더링하는데 사용됩니다. 이 구현은 노드 타입을 LayoutNode로 고정하여 트리의 모든 Layout을 구체화합니다.
    - 상향식으로 노드를 삽입합니다.(새 노드가 트리에 들어올 때 중복 알림을 피하기 위해)
- VectorApplier: 벡터 그래픽을 렌더링하는데 사용됩니다. 이 구현은 노드 유형을 VNode로 고정하여 벡터 그래픽을 표현하고 구체화합니다.
    - 하향식으로 노드를 삽입합니다.
        - 이 방식은 새 노드를 삽입할 때마다 트리의 모든 조상에게 알림을 보내야 합니다.
    - Vector 그래픽의 맥락에서는 어떤 노드에게도 알림을 전파할 필요가 없으므로, 상향식, 하향식 전략 모두 동등한 성능을 보이고 어떤 전략을 택해도 무관합니다.

### 새로운 LayoutNode 구체화 하기

UiApplier 내부 구현을 보면 AbstractApplier<LayoutNode\> 추상체를 상속받고 있으며 내부 함수들은 AbstractApplier 추상클래스의 프로퍼티인 current : T(현재의 타입은
LayoutNode) 에 위임된 것을 확인할 수 있습니다.  
LayoutNode는 순수한 코틀린 클래스이면서, 안드로이드에 대한 의존성이 없습니다. 단지 UI 노드를 모델링하며, 여러 플랫폼(안드로이드, 데스크톱)에 의해 사용된다는 것을 의미합니다.  
LayoutNode는 트리 구조로 연결되어 있어, 각 LayoutNode는 부모 노드에 대한 참조를 가지고 있고 모든 노드들은 동일한 Owner에 연결되어 있습니다.    
여기에서 Owner는 추상화된 개념으로, 플랫폼별로 서로 다르게 구현할 수 있습니다.(플랫폼과의 통합 지점)  
안드로이드에서는 Owner가 View(AndroidComposeView)로, 우리의 Composable 트리(LayoutNode들)와 안드로이드 View 시스템 간의 기본적인 연결점입니다.

- 노드가 연결, 분리, 재정렬, 재측정 등 어떤 업데이트가 발생할 때마다, Owner를 통해 안드로이드 View API를 사용하여 invalidation이 트리거될 수 있으며, 그 결과 최신 변경사항이 다음 그리기
  단계에서 화면에 반영됩니다.
